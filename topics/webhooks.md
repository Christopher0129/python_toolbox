# Webhook 专题

这个专题关注系统如何作为回调提供方或消费方处理异步通知，而不是把它当作普通接口随手实现。

## 常见场景

- 支付结果通知
- Git 仓库事件回调
- 第三方 SaaS 事件推送
- 内部系统异步状态同步

## 实战关注点

| 主题 | 说明 |
| --- | --- |
| 签名校验 | 验证回调来源可信 |
| 幂等处理 | 防止重复投递造成重复业务执行 |
| 超时响应 | 尽快确认接收，重活异步化 |
| 重试策略 | 处理对方重复推送或本方回调失败 |
| 事件追踪 | 能追到某次回调的处理链路 |

## 推荐做法

- 回调入口只做验签、记录、落库、快速确认
- 重型处理放到后台任务或消息系统
- 为事件 ID 建去重键
- 把签名失败、重复消息、处理耗时纳入指标
